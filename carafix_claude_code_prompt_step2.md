## Schema additions

The current schema is mostly right, but three small additions:

### 1. Category-specific fields on jobs

Add these nullable columns to the `jobs` table via a new migration `0002_job_category_fields.sql`:

```sql
alter table jobs
  add column insurance_claim_number text,
  add column warranty_reference text;  -- chassis number or rego, controller types whichever applies
```

Show them conditionally in the form: insurance claim number only when category = `Insurance`, warranty reference only when category = `Warranty`. Private and Dealer don't show either.

### 2. Sequential job numbers

Replace the placeholder `job_number` with a real sequential generator. Migration `0003_job_number_sequence.sql`:

```sql
-- Per-location sequence starting at 100001 each
create sequence if not exists job_number_seq start 100001;

-- Function to allocate next number on insert if not provided
create or replace function set_job_number()
returns trigger as $$
begin
  if new.job_number is null or new.job_number = '' then
    new.job_number := nextval('job_number_seq')::text;
  end if;
  return new;
end;
$$ language plpgsql;

create trigger trg_jobs_set_number
  before insert on jobs
  for each row execute function set_job_number();
```

Job numbers will look like `100001`, `100002`, etc. — plain integers as text, no prefix. The controller never types these; they auto-generate.

### 3. Technician role field

We're going with simplified roles for v1 but want the granular `skills` table preserved for later. Add to `technicians`:

```sql
create type tech_role as enum ('Service Tech', 'Caravan Repairer');

alter table technicians
  add column role tech_role;
```

This is the **display-level** role. The existing `skills` and `technician_skills` tables stay in place untouched — they'll be used in v1.5 when we add granular skill tagging.

For v1 logic, when checking which techs can be assigned to a job, ignore the granular skills table entirely. Either role can take any job; the controller picks who they want. The skills infrastructure is dormant but ready.

## Pages to build

### `/jobs` — Jobs list page

- Table of all jobs at the current location (from LocationContext)
- Columns: Job number, Customer surname, Van make/model, Category, Status badge (coloured by status), Priority badge, Assigned tech, Planned start date, Expected finish date, Flags (small icons for delayed / pickup-ready)
- Top-right: "+ New Job" button → opens NewJobDialog
- Top: filter bar with Status dropdown, Tech dropdown, Category dropdown, search box (matches job number, customer name, rego)
- Click any row → opens Job Detail page at `/jobs/[id]`
- Sort by planned_start_date descending by default

### `/jobs/new` (or modal — your call)

NewJobDialog with these fields, validated with Zod:

**Customer section**
- Customer picker (Combobox): search existing OR "+ Create new customer" inline (small form: name required, phone, email)

**Van section**
- Van picker filtered by selected customer: pick existing OR "+ Add new van" inline (make, model, year, rego)

**Job section**
- Category (radio: Private, Insurance, Warranty, Dealer)
- Insurance claim number (text input, only shown if category = Insurance)
- Warranty reference (text input, only shown if category = Warranty, label: "Chassis number or rego")
- Priority (select: Low, Normal, High, Urgent — default Normal)
- Description (textarea)
- Primary work type (select from these options for now, can refine later):
  - General service
  - Repair
  - Insurance repair
  - Warranty work
  - Pre-purchase inspection
  - Modification
  - Other
- Quoted labour hours (number input, min 0.5, step 0.5)
- Assigned technician (select, filtered to current location's active techs)
- Planned start date (date picker, defaults to next working day)
- Bay (select, optional, from current location's bays)

On submit:
- Create customer if new
- Create van if new
- Insert job (job_number auto-generated by trigger)
- The DB trigger creates the hidden primary task
- Compute and store expected_finish_date = planned_start_date + ceil(quoted_hours/6.5) working days (skip weekends; ignore holidays for now)
- Redirect to `/jobs/[id]` showing the new job

### `/jobs/[id]` — Job Detail page

- Header: Job number large, status badge (clickable to change), customer name, van make/model + rego
- Below header, flag strip: red "DELAYED" badge if `is_delayed`, green "READY FOR PICKUP" if `is_pickup_ready` (from v_job_rollup view)
- Tabs or sections:
  - **Overview**: all job fields editable inline or via edit button; category-specific fields render conditionally; promise date with "Change date" button that requires a reason (logged to promise_date_log)
  - **Parts**: table of parts on this job, "+ Add part" button → AddPartDialog (description, supplier, quantity, is_critical toggle, status, ordered_date, eta_date), inline edit, mark received
  - **History**: combined timeline of job_status_log and promise_date_log entries, most recent first
- Status change: dropdown of statuses with an optional reason field; on save inserts into job_status_log via the existing trigger
- Delete button (bottom, with confirm) — for testing, will be restricted by RLS later

### `/technicians` — Technicians page

- Table: Name, Role (Service Tech / Caravan Repairer), Location, Email, Active toggle
- "+ Add technician" → form (name, email, role, location, productive_hours_per_day default 6.5, weekly_capacity_hours default 32.5)
- Edit inline or via row click
- Soft-delete via `active = false`, don't hard-delete (preserves history)

### `/customers` — minimal Customers page

- Read-only table: Name, Phone, Email, count of vans, count of jobs
- Created inline via New Job form; no standalone "Add Customer" flow yet
- Click a row → simple detail view showing their vans and job history

### `/parts` — workshop-wide parts view

- Across all jobs at current location, list parts where status in ('Needed', 'Ordered')
- Columns: Job number, Customer, Part description, Supplier, ETA date, Status, Days overdue (if ETA passed)
- Sort by ETA ascending
- Quick action: mark as Received → updates received_date, sets status to Received, refreshes the parent job's status if it was Waiting on Parts
- Read-only otherwise; full editing happens on the Job Detail page

## RLS policies — finally turn them on

The schema has RLS enabled but no policies. Add them now via migration `0004_rls_policies.sql`:

- Controllers (role='controller') can `select`, `insert`, `update`, `delete` on all rows in their `default_location_id` for: jobs, tasks, parts, job_attachments, job_status_log, promise_date_log
- Controllers can `select` and `update` on technicians in their location, `insert` new ones
- Controllers can `select`, `insert`, `update` on customers and vans (no location filter — shared across the business)
- Controllers can `select` everything on locations, skills, bays, holidays (read-only reference data for now)
- Block all access for unauthenticated users
- Technicians table policies: a tech can `select` their own row only (for the v0.2 mobile view) — but no technician auth users exist yet, so this is dormant

After applying, **test the policies work** by querying as the seeded controller and confirming Arundel/Currumbin filtering takes effect.
